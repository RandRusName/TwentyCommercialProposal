# ADR-001: CommercialProposal Aggregate With Items and Stages

## Status

Accepted.

## Context

The current app can generate XLSX/PDF documents from a `CommercialProposal`, but legacy schema `1.0` may synthesize content from a short Opportunity card. Real proposals require persisted work items, stages, terms, and deterministic totals.

Prompt 5.1 must add the aggregate model while preserving legacy generation until Prompt 5.3 implements schema `2.0`.

## Decision

Use `CommercialProposal` as the aggregate root:

```text
CommercialProposal
├── CommercialProposalItem[]
└── CommercialProposalStage[]
```

Add `CommercialProposal.contentModelVersion`:

```text
LEGACY_V1
AGGREGATE_V2
```

Amount semantics are model-version dependent:

```text
LEGACY_V1.amount    = legacy snapshot / historical value
AGGREGATE_V2.amount = SUM(CommercialProposalItem.lineAmount)
```

Store `CommercialProposalItem.lineAmount` physically, but calculate it only server-side.

Use aggregate save:

```text
POST /commercial-proposals/:id/save-editor
```

Make aggregate save replay-safe through:

- child `clientKey` UUID;
- save `operationId` UUID;
- proposal `lastEditorOperationId`;
- parent+clientKey lookup/upsert.

Use `editorRevision` for optimistic concurrency. Prompt 5.1 must spike Twenty SDK/Core API v2.20.0:

- if CAS is supported, use atomic compare-and-set for final proposal update;
- if CAS is not supported, document and implement best-effort optimistic concurrency.

Before Prompt 5.3:

- `LEGACY_V1` generation remains schema `1.0`;
- `AGGREGATE_V2` generation is blocked with `COMMERCIAL_PROPOSAL_GENERATION_MODEL_NOT_SUPPORTED`.

After Prompt 5.3:

- `LEGACY_V1` routes to schema `1.0` / template v1;
- `AGGREGATE_V2` routes to schema `2.0` / template v2.

Defer `CatalogItem` and `CommercialProposalItem.catalogItem` to Prompt 5.4.

## Consequences

Positive:

- Legacy records remain safe and readable.
- New aggregate content is not accidentally generated by the legacy synthetic generator.
- Retry after partial save is convergent and avoids duplicate children.
- `amount` remains a single visible total field while preserving legacy semantics.

Negative:

- Prompt 5.1 must carry a transition guard until Prompt 5.3.
- Strict concurrency depends on Twenty platform CAS support.
- Without compound unique index support, parent+clientKey uniqueness is best-effort at the application layer.

## Alternatives Considered

### Use `version` Instead of `contentModelVersion`

Rejected. `version` is the business version of the proposal. `contentModelVersion` is the format of stored content.

### Generate AGGREGATE_V2 Through Legacy Generator Temporarily

Rejected. It would silently discard real line items/stages and could produce incorrect totals.

### Add `totalAmount`

Rejected. It creates long-term duplicate total fields. `amount` is reused with model-version-qualified semantics.

### Add CatalogItem in Prompt 5.1

Rejected. Items must first work as independent snapshots. Catalog is a later acceleration layer.

## Implementation Notes for Future Prompts

- Do not change existing universal identifiers.
- Do not add `catalogItem` relation in Prompt 5.1.
- Keep `DRAFT` and `FAILED` editable.
- Keep generated/sent/accepted proposals read-only.
- Header-only save on `LEGACY_V1` must not convert or overwrite amount.
- First successful save with valid items converts irreversibly to `AGGREGATE_V2`.
- Use schema/template mismatch rejection in generation.
